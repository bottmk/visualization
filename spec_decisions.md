## A-1. 高さマップの物理スケール定義

**問題**: `get_height_map()` の戻り値にグリッドサイズと物理ピクセルサイズが未定義だった。FFTの周波数軸計算に両方が必須。

**採用**: `HeightMap` dataclass（`data: np.ndarray` + `pixel_size_um: float`）+ `get_preview_height_map()` の2モード方式

**却下した選択肢**:
- `pixel_size_um` + `grid_size` のみ: スケール情報が配列と分離するため渡し忘れリスクあり
- `physical_size_um` + `grid_size`: FFTの基本量は `pixel_size_um` であり直感に反する
- `mode='full'/'preview'` の二択: 中間解像度（128/256等）に対応できない

**プレビューモードの設計経緯**:

当初「マルチ解像度」と呼んでいたが、`pixel_size_um` 固定で `grid_size` を変えるのは物理面積が変わるだけで解像度は変わらないため不正確。以下の2モードを両実装することで合意した。

| モード | 固定値 | 変化する値 | 散乱角上限 | ヘイズ確認 |
|---|---|---|---|---|
| `reduced_area`（A） | pixel_size_um=0.25μm | grid_size縮小 | 90°（維持） | 可 |
| `reduced_resolution`（B） | physical_size_um=1024μm | pixel_size_um拡大 | 約7.9° | 不可 |

**ヘイズと広角散乱の関係**: ヘイズは2.5°以上への散乱光の割合であり、散乱角上限を維持する `reduced_area` モードが必要。`reduced_resolution` モードではヘイズ計算が不可能であるため、DynamicMapのデフォルトは `reduced_area` とする。

**本計算パラメータ**: `grid_size=4096`, `pixel_size_um=0.25μm` → 物理サイズ 1024μm

## A-2. 位相変換の数式定義

**問題**: 反射・透過の位相変換式が未定義で実装不能だった。

**採用**: 斜入射補正あり・偏光なしのスカラー式（選択肢2）

```
BRDFモード: φ(x,y) = (4π/λ)·n₁·h(x,y)·cosθ_i
BTDFモード: φ(x,y) = (2π/λ)·(n₂·cosθ_t - n₁·cosθ_i)·h(x,y)
斜入射傾き: φ_tilt(x) = (2π/λ)·n₁·sinθ_i·x
```

**FFT法とスカラー近似・偏光の関係**:

レビュー中に「フレネル係数 r_s, r_p を U(x,y) に乗じれば偏光を扱える（選択肢3）」という案を検討したが、以下の理由で却下した:

- `U(x,y) = F_pol · exp(iφ)` は選択肢2 + 定数フレネル補正に過ぎない
- 厳密な偏光依存は各点の局所法線ごとにフレネル係数が変わるため `F_pol(θ_local(x,y))` が必要
- これはスカラー近似の範囲を超えたベクトル波動光学であり、FFT法の範疇ではない
- **FFT法では偏光を扱わない**。偏光依存BSDFはPSD法のQ因子（Appendix A）に委ねる
- Section 3.2の「偏光指定＋広角 → 警告」はこの理由から発生している

## A-3. PSD法の偏光因子 Q の定義

**問題**: Rayleigh-Rice理論の偏光因子Q（S/P/Unpolarized、BRDF/BTDF）が未定義だった。

**採用**: 完全形（Elson-Bennett形式）をデフォルト、`approx_mode=True` で簡略形（Stover近似）に切替可能。定義はAppendix Aに記載。

**選択肢の整理**:

| | BRDF の Q 式 | BTDF の Q 式 |
|---|---|---|
| 簡略形（Stover） | θ_i のみでフレネル評価 | 簡略形＋エネルギー保存係数 E |
| 完全形（Elson-Bennett） | θ_i と θ_s 両方でフレネル評価 | θ_i と θ_t 両方でフレネル評価 |

**「θ_s = 0 で簡略形になるか」の検証結果**:

ユーザーから「完全形に θ_s=0 を代入すれば簡略形になるか」という質問があった。具体例で検証した結果、一致しないことが判明した。

- 完全形（θ_s=0 代入）: Q_s ≈ 0.142（n₁=1.0, n₂=1.5, θ_i=45° の場合）
- 簡略形: Q_s = |r_s(θ_i)|² ≈ 0.154

両者は**異なる近似仮定から導出**されており、θ_s=0 の代入によって完全形から簡略形を導出することはできない。簡略形は「θ_s が小さい（近軸近似）」を**導出過程の中途で**適用した別の式である。

このため `approx_mode` パラメータによる明示的な切り替えとした。

**AGフィルムの主用途（透過・BTDF）があるためBRDF/BTDF分離が必須**と判断し、付録方式を採用。

## B-1. Log スケール誤差計算のゼロ処理

**問題**: `log10(0)` や測定ノイズフロア以下の値の処理が未定義だった。

**採用**: 固定フロア + マスク処理の組み合わせ（選択肢4）

```python
mask = measured > bsdf_floor
log_rmse = np.sqrt(np.mean(
    (np.log10(np.maximum(simulated[mask], bsdf_floor))
   - np.log10(measured[mask])) ** 2
))
```

**設定**: `bsdf_floor` はデフォルト `1e-6 sr⁻¹`（散乱計の典型的ダイナミックレンジ 120dB に対応）。`config.yaml` で装置ごとに変更可能。Optuna セッション中は固定して試行間比較の公平性を保つ。

**却下した選択肢**:
- 実測データの最小値からフロアを自動決定（選択肢2）: 実験ごとにフロアが変わり試行間比較が不公平になる
- マスクのみ（選択肢3）: シミュレーション値がゼロになった場合に log10 が破綻する

## B-2. Adding-Doubling 法の離散化パラメータ

**問題**: 散乱行列の方向基底の離散数（ガウス求積点数・フーリエモード数）が未定義で計算コストが見積もれなかった。

**採用**: プリセット + 個別オーバーライド方式（選択肢4）

**装置スペックから導出したデフォルト値**:
- 装置: 天頂角 1°刻み（90点）、方位角 10°刻み（36点）
- `standard` プリセット: `n_theta=128`（ガウス求積の非一様性を考慮し2倍余裕）、`m_phi=18`（方位10°のナイキスト限界）

```
fast:     n_theta=32,  m_phi=8    ← DynamicMap プレビュー用
standard: n_theta=128, m_phi=18   ← 装置分解能に対応（デフォルト）
high:     n_theta=256, m_phi=36   ← 2倍オーバーサンプリング
```

Adding-Doubling の出力は装置の測定グリッド（天頂1°、方位10°）へ補間してから実測データと比較する。

## B-3. ギラツキ計算のディスプレイ光学系パラメータ

**問題**: ギラツキ計算に必要な観察条件・ディスプレイ仕様・照明条件が未定義だった。

**採用**: 全グループをプリセット選択 + 数値入力の両方に対応。グループ4（積分・指標）は固定。

| グループ | 内容 | 設定方法 |
|---|---|---|
| 1. 観察条件 | 観察距離、瞳孔径 | プリセット or 数値入力 |
| 2. ディスプレイ仕様 | 画素ピッチ、サブピクセル配列 | プリセット or 数値入力 |
| 3. 照明条件 | 波長（単波長/RGB） | プリセット or 数値入力 |
| 4. 評価指標 | 1画素積分、Cs=σ/μ | 固定 |

プリセット優先ルール: 数値が明示された場合（null でない）は数値を優先。

## C-1. 物理単位系の統一

**採用**: μm（マイクロメートル）統一（選択肢2）

AGフィルム用途（nm〜μmスケール）で数値が扱いやすい（`pixel_size=0.25`, `wavelength=0.55`, `Lc=2.0` 等）。SI基本単位（m）では数値が `2.5e-7` 等になり可読性が低い。用途別混在（選択肢3）は変換ミスのリスクがある。

外部入力の変換例: 波長 [nm] → [μm] は ÷1000。表示用に nm 換算ユーティリティを用意する。

## C-2. 依存パッケージのバージョン管理

**採用**: `pyproject.toml` に範囲指定 + `requirements.lock` に完全固定の二層構造（選択肢3）

HoloViews・Panel は破壊的変更が多く、Optuna・MLflow も API 変更が活発なため、CI/CD では完全固定が必要。開発中は範囲指定で柔軟に対応。

## C-3. 入出力インターフェース

### C-3a. 設定ファイル形式
**採用**: YAML（選択肢1）。コメントで設定の意図を記述でき、研究用途に向く。

### C-3b. CLI 設計
**採用**: サブコマンド型（選択肢1）。`simulate / optimize / visualize / report` の4コマンド。Optuna・MLflow の CLI 設計と一貫性がある。

### C-3c. Parquet スキーマ
**採用**: long format（`bsdf` 列 + `method` カテゴリ列）

当初は wide format（`bsdf_fft`, `bsdf_psd`, `bsdf_measured` を別カラム）も検討したが、新しい計算手法（例: RCWA）を追加する際にスキーマ変更が不要な long format を採用。HoloViews での `groupby('method')` によるオーバーレイ描画が自然に対応できる。

同一条件の手法間比較は `pivot` 操作で対応する。

**実測データの生ファイルフォーマット**: 装置のヘッダ情報が含まれるため、実ファイル提供後にローダー仕様を追記する。

## C-4. DynamicMap パフォーマンス設計

**採用**: LRUキャッシュ + ドラッグ中間引き計算の組み合わせ（選択肢4）

**3段階の解像度**:

| フェーズ | grid_size | 目安時間 | 用途 |
|---|---|---|---|
| ドラッグ中 | 128 | ~0.1秒 | リアルタイム応答 |
| ドラッグ停止後 | 512 | ~0.6秒 | パラメータ確認 |
| 本計算 | 4096 | ~数分 | Optuna最適化用 |

`RandomRoughSurface` 等の乱数を使うモデルは `random_seed=42` でプレビュー用シードを固定してキャッシュを有効にする。本計算ではシードを変えてアンサンブル平均を取る。

キャッシュのみ（選択肢2）ではドラッグ中に重いため、Panel の `throttled` 機能による間引きを組み合わせる。

---

## C-5. Optuna 重複試行スキップの類似度定義

**採用**: 正規化ユークリッド距離（選択肢2）

各パラメータを探索範囲 [low, high] で 0〜1 に正規化後、ユークリッド距離を計算。閾値は `distance_threshold: 0.01`（`config.yaml` で変更可能）。

**却下した選択肢**:
- 各パラメータの絶対差（選択肢1）: プラグイン形状モデルが増えるたびに閾値追加が必要
- 重み付きユークリッド距離（選択肢3）: BSDF感度が事前にわからない段階で重みの設定が困難。Optunaの探索で自然に収束するため重み付けは不要
- Optuna pruner に委ねる（選択肢4）: BSDF計算は途中評価が難しくprunerの恩恵を受けにくい。目的（重複除去）も異なる
